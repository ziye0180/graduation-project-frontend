# 第二部分：课题背景与意义

---

## 第3页：课题背景

### 一、科研工作中的文献管理痛点

#### 1. 信息管理混乱

| 问题类型 | 具体表现 |
|----------|----------|
| **分散存储** | 文献分散在本地、云盘、邮箱等多处，查找困难 |
| **手工管理** | Excel记录效率低、易出错，无法全文检索 |
| **版本混乱** | 团队成员使用不同版本资料 |

#### 2. 团队协作困难

- 缺乏项目与文献的关联机制
- 无法追踪谁上传了哪些文献
- 缺乏权限控制（成员/管理员/负责人）

#### 3. 现有工具不足

| 工具类型 | 代表产品 | 不足之处 |
|---------|---------|---------|
| 通用文献管理 | EndNote、Zotero | 缺乏项目协作功能 |
| 云存储工具 | Dropbox、OneDrive | 无文献元数据管理 |
| 项目管理工具 | Notion、Trello | 不专注科研场景 |

### 二、数字化管理的迫切需求

```
传统方式                    迫切需求
────────                    ────────
📁 本地文件夹        ➜      ☁️ 云端集中管理
📝 Excel记录         ➜      🗄️ 结构化数据库
👤 个人使用          ➜      👥 团队协作共享
🔍 文件名搜索        ➜      🎯 全文检索+分类
```

## 第4页：研究意义

### 一、理论意义

- **领域驱动设计（DDD）实践**：明确User、Literature、Project三大领域边界
- **UML系统化建模**：用例图、时序图、活动图、类图完整设计
- **分层架构设计**：Controller → Service → Logic → DAO，降低耦合

### 二、实践意义

| 优化维度 | 传统方式 | 系统方式 | 效率提升 |
|---------|---------|---------|---------|
| 文献检索 | 手动翻文件夹 | 关键词搜索 | **节省80%时间** |
| 信息录入 | 手动Excel | 结构化表单 | **减少70%错误** |
| 团队共享 | 微信/邮件 | 项目文献关联 | **即时同步** |

### 三、技术意义

**前端技术栈**
- React 19 + Vite 7 + TypeScript 5.9
- TanStack Router/Query/Table
- Shadcn UI + Tailwind CSS 4
- Zustand + React Hook Form + Zod v4

**后端技术栈**
- Spring Boot 3.5.7 + Java 17
- MyBatis Plus 3.5.14
- Spring Security + JWT (JJWT 0.12.6)
- BCrypt密码加密
